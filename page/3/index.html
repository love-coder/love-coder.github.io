Web开发总结(一年)之业务开发的思考

发表于 2017-03-06 | 分类于 阶段总结 |

本篇文章是《Web开发总结(一年)》系列的业务开发思考篇。文中主要是对我在过去一年参与的一些项目开发经历的整理和思考。这个模块是四个模块中写的最纠结的。因为业务开发不同于很多纯技术开发，如判断图像识别算法的好坏，可以在实际应用场景下跑几个不同程度的Case以及理论推导算法复杂度就可以。但业务开发很难有一个可以量化的指标告诉我们，哪些业务设计是好的？但是不是项目开发就没有套路了呢？

随机应变就是套路！而随机应变的前提是对业务整体有足够了解。如果对业务不了解，容易陷入只见树木不见森林的尴尬境地。也就是说，即使对于自己负责的业务模块设计的再精妙，代码写的再优美，也逃不过后期需要被重构的分险。很多时候，重构的成本并不低，甚至比重新开发还要高。所以业务开发者必须首先对业务整体有感知，知道该业务的哪些模块是不会轻易变化的，而哪些模块又是容易变化的。对于容易变化的点，要预设足够的设计空间，便于项目迭代过程中的合理开发。同时，对于需要跨团队合作项目，开发者一定要在开发前先和其他团队把职责划分清，避免相互间代码重复、调用关系不清等情况。

Web开发总结(一年)概述

发表于 2017-03-06 | 分类于 阶段总结 |

这一年最大的感触就是说话小心谨慎了，不会轻易去点评一门技术，更愿意多花点时间去了解这些技术背后的东西，尤其是它的演变过程和应用场景。太阳下没有新鲜事物，而这些新技术往往是由当时旧的技术在解决某个或某些问题时代价较高，主动或被动演变而来。有些新技术表面上牺牲了一定的灵活性，但是开发效率却大大提高了；有些新技术在一些人看来可能无法理解，但是如果结合历史背景和具体应用场景，也许原本“不堪”的设计才是最好的设计。

这一个月都在整理过往的笔记。期间打开了某机场售票项目代码，那是我第一个独立负责的项目，看了几眼其中的代码，顿时也哭笑不得。喜的是，能知道过去的不足，说明这段时间进步了。忧的是，当时尽管已经有意识从包管理、类命名、变量命名等角度去规范代码，但还停留在人云亦云的角度，比如将大量的业务逻辑写在了Controller里面，而这原本应该是很薄的类。这带来的直接坏处就是，后面开发其他模块时，没法直接复用业务代码，本质上就是还没有意识到抽象出Service层。是不是抽象出Service层就好了？带着这些问题，用了半个月闲暇时间断断续续整理出一年Web开发总结的系列。该系列主要包括：《对Servlet规范的思考》、《对Web MVC框架的理解》、《对服务化架构的理解》以及《对业务开发的思考》四篇组成。

近期将更新文章

发表于 2017-02-28 | 分类于 工作随记 |

写完这几篇技术文章之后发现，写文章的难度远超过我当初的想象。不仅仅要把问题说明白，同时为了和其他类似文章有所区别，还需要研究下底层原理，好融会贯通，沉淀基础。而光是将问题说明白难度已不小，这需要能够比较全面的看待问题，并给出切实可行的解决方案。同时，每天都有新技术出现，若一直追寻这些技术，则容易陷入各种困境。可喜的是这些新技术的底层原理是相似的，这就是为什么除了把事情说明白之外，还要在必要时深入去了解底层实现。这样，在下次遇到新的技术时，可以类比学习，触类旁通，并保持对技术的热情。
