Web开发总结(一年)

发表于 2017-02-16 | 分类于 随记 |

这一年最大的感触就是说话小心谨慎了，不会轻易去点评一门技术，更愿意多花点时间去了解这些技术背后的东西，尤其是它的演变过程和应用场景。太阳下没有新鲜事物，而这些新技术往往是由当时旧的技术在解决某个或某些问题时代价较高，主动或被动演变而来。有些新技术表面上牺牲了一定的灵活性，但是开发效率却大大提高了；有些新技术在一些人看来可能无法理解，但是如果结合历史背景和具体应用场景，也许原本“不堪”的设计才是最好的设计。

这一个月都在整理过往的笔记。期间打开了某机场售票项目代码，那是我上线的第一个独立负责的项目，看了几眼其中的代码，顿时也哭笑不得。喜的是,能知道过去的不足，说明进步了。忧的是，当时尽管已经有意识从包管理、类命名、变量命名等角度去规范代码，但还停留在人云亦云的角度，比如将大量的业务逻辑写在了Controller里面，而这原本应该是很薄的类。这带来的直接坏处就是，后面开发其他模块时，没法直接复用业务代码，本质上就是还没有意识到抽象出Service层。是不是抽象出Service层就好了？带着这些问题，用了几天时间陆续整理出这篇总结。

Web是什么

早期的Web应用，可以视为是基于HTTP 1.0协议的静态应用。简单理解就是，用户通过在浏览器输入一串URL，就可以看到网页，并可以通过超链接实现不同页面的跳转。比如，我们可以用下面的Java编码实现上述功能：通过在本地监听在80端口，并根据不同用户访问的参数不同（这里是GET方式直接写在URL中）回复不同的内容。当然我只是简单示意，一般意义上静态页面技术还包括CSS和JS。它们关系可以简单理解为，HTML是内容，CSS是装饰，JS是动作（效果）。

如果以现在的角度来看，上述实现存在什么问题？

页面内容一层不变，纯静态（也不一定不好，视具体场景定，比如博客）

需要自己实现管理HTTP连接

路由不够灵活

耦合严重

重复代码多

线程不复用

同步阻塞严重，支持并发量小
