Web开发总结(一年)概述

发表于 2017-03-06 | 分类于 阶段总结 |

这一年最大的感触就是说话小心谨慎了，不会轻易去点评一门技术，更愿意多花点时间去了解这些技术背后的东西，尤其是它的演变过程和应用场景。太阳下没有新鲜事物，而这些新技术往往是由当时旧的技术在解决某个或某些问题时代价较高，主动或被动演变而来。有些新技术表面上牺牲了一定的灵活性，但是开发效率却大大提高了；有些新技术在一些人看来可能无法理解，但是如果结合历史背景和具体应用场景，也许原本“不堪”的设计才是最好的设计。

这一个月都在整理过往的笔记。期间打开了某机场售票项目代码，那是我第一个独立负责的项目，看了几眼其中的代码，顿时也哭笑不得。喜的是，能知道过去的不足，说明这段时间进步了。忧的是，当时尽管已经有意识从包管理、类命名、变量命名等角度去规范代码，但还停留在人云亦云的角度，比如将大量的业务逻辑写在了Controller里面，而这原本应该是很薄的类。这带来的直接坏处就是，后面开发其他模块时，没法直接复用业务代码，本质上就是还没有意识到抽象出Service层。是不是抽象出Service层就好了？带着这些问题，用了半个月闲暇时间断断续续整理出一年Web开发总结的系列。该系列主要包括：《对Servlet规范的思考》、《对Web MVC框架的理解》、《对服务化架构的理解》以及《对业务开发的思考》四篇组成。

Web到底什么

早期的Web应用，可以视为是基于HTTP 1.0协议的静态应用。简单理解就是，用户通过在浏览器输入一串URL，并将请求参数使用GET或POST等方式提交服务端。服务端根据这些不同请求，调用相应的方法，提供文件上传下载服务，网页和在线视频等。比如，我们可以借助Java语言简单编码实现上述网页功能：通过在本地监听在80端口，并根据不同用户访问的参数不同（这里是利用GET方式且直接写在URL中）回复不同的内容。只是简单示意，一般意义上，静态页面技术还包括CSS和JS。它们关系可以简单理解为，HTML是内容，CSS是装饰，JS是动作（本文重点是以网页服务为主，流媒体服务和文件服务等其他类型服务暂时不予涉及）。

SimpleServer类简单实现了解析HTTP请求，并根据不同的请求参数调用不同方法。比如如果用户请求名为say的服务，那么SimpleServer将取出name的具体参数值，并返回渲染后的页面。

上述实现可能存在什么问题？

需要自己实现管理HTTP连接

管理HTTP连接和解析HTTP各种参数等工作并不容易，尤其是在业务复杂、访问量比较大的时候。而这应该是网站开发尤其进行业务开发时需要关注的吗？如果这些代码每次都需要自己实现，无疑浪费了大量时间进行的“非核心模块”（相对于大部分开发目的本身而言）的开发上。
有解决的办法没？有，那就是标准化，遵从Servlet规范。所以看来Servlet也不是与生俱来的技术，而是人们经过一次又一次重复且痛苦的经历之后才被接受。而Servlet容器比如Tomcat就有Connect模块，用来管理浏览器的连接请求，并分发到相应的处理模块以及返回最终页面。它会在每次连接建立时，初始化HttpServletRequest类并提供获得具体请求方法、远程端口、编码、HTTP头字段等方法。除此以外，针对HTTP连接的无状态性，Tomcat还提供Session、Cookie管理等基本功能。这样就业务开发者就可以从HTTP管理工作中彻底解放，以此专注业务开放本身。其实很多技术的演变也是遵循这样的思路，它们将基础模块抽象出来做到可灵活配置（某种程度上，一种框架或技术是另一个框架或技术的基础组成，比如RPC框架HSF底层利用NIO框架Netty通信，利用Spring IOC管理Beans等）；将开发的重点集中到跟自己切身相关的模块上，如利用Spring MVC开发Web只需编写相应的Controller即可完成逻辑处理和页面处理，至于怎么封装参数、怎么路由、怎么渲染页面等很重要但不是应该关注的点，那就交给框架去做。

路由不够灵活

SimpleServer类中resolve方法，可以根据不同的请求URL判断进入不同的处理模块。但如果新加一个模块呢？继续修改原来代码，增加 if else判断逻辑？或是统一写在配置文件中，具体使用时去读取？前者显然不行，后者实际上是纯用Servlet开发，也正是把映射关系写在web.xml中（Servlet3.0以后支持@annotation机制，但本质没有改变，代码编译后关系被保存）。在业务不多时可以胜任，但当业务变得越来越多且更复杂时，灵活以及高效维护这些映射关系就变得尤为重要。以及如何根据请求的方法调取相应的处理Handler，是Demo中那种硬编码的方式，还是基于反射技术invoke相应方法还或是其他？

页面内容一层不变，纯静态（也不一定不好，视具体场景定，比如博客）

Demo中只是提供了纯静态的展示，不涉及数据库相关操作。这对于早期的新闻类网站或者博客类网站是比较合适的。但一旦想提供多元化的服务的网站，比如电商、社交等，那几乎就不可行。如果需要呢？代码怎么被合理组织？怎么连接数据库，是直接通过JDBC？连接线程怎么管理，用池化技术？

耦合和冗余严重

SimpleServer类用了100行左右代码实现了一个最基本的网站服务，但我们也看到它将HTTP连接、参数解析、具体业务处理以及页面渲染等写在一起。这在项目变大后，几乎没法维护，因为不同类型的模块间耦合和冗余非常严重。模块解耦是Web开发甚至其他所有开发都必须面对的问题。文章后面要谈的前后端分离、MVC框架、各种服务化架构以及中间件某种程度上都在一些地方做了解耦。一方面使得代码变得清晰（虽然有时会增加很多“无效”代码）、易于维护、方便多团队并行开发，也使得开发者专注于“核心业务”变得可能。代码冗余使得相似代码出现在多处，造成项目维护不便。冗余的本质是没有合理抽象出变化点，且没有对这些变化点进行封装成为基础模块。在业务开发时，很多工程师在抽象出Service层之后又继续抽象出一层甚至几层CoreService层也是做得封装变化点工作，起到复用代码的目的。

线程不复用

回答为什么线程不复用之前，我们不妨研究下线程到底是什么？和进程是什么关系？如何复用线程？复用本质又是什么？
补充点：线程和进程

同步阻塞严重，支持并发量小

代码中的BIO方式同步性较好且简单易用。但针对一些费时操作比如文件IO、网络IO以及数据库操作等，一直阻塞等待处理结果必然导致同一时刻支持的连接数是有限的。NIO方式便应运而生。

单机应用

这种编写方式只适合在单机部署。一味提高单机性能一来是性价比不高，二来是不可能能够持续应对高访问需求和稳定性要求。服务器集群成了必须要面对的问题。但紧接着又会带来一些新的问题，如Session共享、节点通信、单点故障、分布式事务以及容灾备份等。

Java Web开发基础

现在几乎很少有项目直接利用Servlet做开发，但这并没有降低Servlet在Java Web开发中的地位，相反在某些时候重要性更加突出了。正如我刚开始写Web时，就是基于SSH框架做开发。业务方提出新需求，我就新添加相应的XxxController、XxxDAO和它的实现类XxxDAOImpl，如此竟也完成了项目第一版本并成功上线。后来，不断的有新业务被提出，如此添加Controller和DAO导致代码冗余和耦合问题越来越严重，甚至有异常订单出现。

那段时间整天想着，怎么把代码写的健壮一些，好面对各种突发情况，怎么抽象出变化点让冗余代码少点，怎么榨干单机性能把并发数提高等等。庆幸在那段时间看了很多基础的东西，比如Servlet规范、Spring IOC、Spring AOP、Spring MVC以及Tomcat原理等。后来突然发现，原来Web开发就是基于HTTP/HTTPS协议开发一个网络服务，所谓了一堆新技术也无非在某些点上封装变化点和解耦模块。那些层出不穷的框架是帮助开发者更加专注于他们核心模块的开发，而减少不必的非核心模块代码编写。比如Spring MVC就是在Servlet基础上又做了一层封装，避免了书写大量的Servlet中重复代码，轻轻的几行就可以实现路由配置和页面跳转功能。它变了吗？没有！底层运行的依旧是一个个Servlet。
